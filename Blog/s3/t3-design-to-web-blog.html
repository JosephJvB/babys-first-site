The design process

  • What is a responsive site? Why is responsiveness important?

    • A website is responsive if it is designed to be used on a range of screen sizes. What this means is when the browser detects a different screen size, the website will respond by rearranging its components so that it is still readable and accessable at the different screen size. Responsiveness is important to think about when building a website because there is no standardized screen size for browsers so as a designer you need to make sure that the site will look good whether a user is accessing your site from a phone, a tablet or a desktop PC. Also I know I like to say halve my web browser to work on a local program but still view website info. A nice reponsive site will make my life easy in that situation.

  • What mobile first design is, and why it's important.

    • Mobile first design is the idea that when you build a website, you code your site to fit the screen resolution of a mobile screen (~400px). This is done by having your primary style rules (CSS) be suited to the mobile screen size, and then have extra rules apply when the screen gets larger. Mobile first design is good because it means when you view a mobile screen, the mobile has to 'parse' less code to load so it puts less strain on the mobile's processing - enabling a better user experience. Typically PC's and laptops will be more powerful than a handheld device and won't have trouble parsing the extra CSS code.

  • What frameworks are, and their pros and cons.

    • Frameworks are pre-written sections of code that you can use to build your own website. They're good because it saves you the time of writing out lots of basic rules. Frameworks are good also as it means whatever code you write will be less cluttered. One con I can think of about frameworks is that if you include a framework that you didn't fully read (or understand) you may run into problems such as code behaving strangely and not knowing the root of the problem if that problem is tied up in a framework somewhere.

  • What a wireframe is and why we use it.

    • A wireframe is a visual plan of a websites layout. We use wireframes to break the website down into its box-elements and arrange them. Once you decide the purpose of a website, you use the wireframe to decide how it will achieve that purpose. Do you want to user to read a body of text? Plan to have a easily readable section for a text element, etc.

  • The aspects of your wireframes you found difficult to implement, and why.

    • The plans I made at the wireframe stage didn't end up being an exact prediction of how my site has finished up. I wanted to create my own media queries with the display:none rule/value set to have elements hidden at the moblie size that would then come in as the screen got bigger. But I found that the media queries in the skeleton framework actually had a pretty satisfactory effect - saving me the hassle of creating my own code. I did have a go at trying it but I couldnt get my head around it completely. It's definitely something I'm keen on figuring out though. Also I created some scroll overflows, as i had planned in the wireframes. But they dont work the way I'd hoped. I think it's a matter of setting a boundary size in the CSS but it doesn't obscure the content too badly so I'm happy to leave it and tinker with it later.
